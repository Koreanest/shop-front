# 프론트 팀원 워크오더 재정리

## 1. 최우선: 결제 흐름 추가

지금 가장 큰 남은 기능은 이거다.
현재는 **주문 생성까지만 되고, 결제 단계가 빠져 있음**.

### 작업 목표

* 주문 생성 후 바로 끝내지 않고 **결제 단계로 연결**
* 주문 상세에서도 **결제하기 버튼** 노출
* 결제 성공 시 주문 상태 반영
* 결제 실패/취소 시 사용자 메시지 처리

### 프론트 작업 범위

#### A. 결제 진입 방식 정리

둘 중 하나로 화면 흐름 확정:

1. **checkout에서 주문 생성 → 결제 페이지로 이동**
2. **주문 상세에서 미결제 주문에만 결제 버튼 노출**

현재 화면 기준으로는
**주문 생성 후 주문 상세로 이동하고, 상세에서 결제하기 버튼 노출**이 제일 자연스럽다.

#### B. 주문 상세 페이지 수정

* 상태가 `CREATED` 또는 `주문접수`일 때만 **결제하기** 버튼 표시
* `CANCELLED`, `PAID`, `SHIPPED` 등에서는 숨김 또는 비활성화
* 지금 있는 `주문 취소` 버튼과 함께 조건 분기

예시 조건:

* 주문접수: 결제하기 + 주문취소
* 결제완료 이후: 결제하기 숨김, 주문취소도 정책 따라 제한
* 주문취소: 둘 다 숨김

#### C. 결제 API 연결 준비

백엔드 스펙 확정 필요 항목:

* 결제 요청 엔드포인트
* 결제 성공 후 주문 상태값
* 결제 실패 응답 구조
* 테스트 결제 리다이렉트 방식 또는 SDK 방식

프론트는 이걸 기준으로 작업:

* `payOrder(orderId)` API 함수 추가
* 버튼 클릭 시 결제창 or 결제 요청 호출
* 성공 시 상세 재조회 또는 상태 갱신
* 실패 시 에러 메시지 표시

#### D. 결제 상태 UI

주문 상태 문구를 사용자용으로 분리:

* CREATED → 주문접수
* PAID → 결제완료
* PREPARING → 배송준비중
* SHIPPED → 배송중
* COMPLETED → 배송완료
* CANCELLED → 주문취소

## 2. 주문 상세 페이지 보강

스크린샷 기준으로 여기 보강 포인트가 보인다.

### 현재 확인된 문제

주문 상품 영역에

* 주문가격: 0원
* 합계: 0원

이게 뜨고 있다.

이건 프론트든 백엔드든 **필드 매핑 문제** 가능성이 높다.
프론트 팀원 작업으로는 먼저 이걸 확인해야 한다.

### 점검 항목

* 주문 상세 응답에서 `unitPrice`, `lineTotal`이 실제로 내려오는지 확인
* 프론트가 `price`, `totalPrice`, `lineAmount` 같은 다른 이름을 읽고 있지 않은지 확인
* 목록 총금액은 정상인데 상품 단가/합계만 0이면 **상세 item 필드명 불일치** 가능성이 높음

### 수정 목표

주문 상세 상품 카드에 아래가 정상 표시되도록 수정

* 상품명
* 사이즈
* SKU
* 주문가격
* 수량
* 합계

## 3. 주문 상태별 액션 분기 정리

지금은 주문 취소는 들어가 있지만, 상태별 버튼 정책을 프론트에서 정리해야 한다.

### 상태별 화면 정책

#### 주문접수

* 결제하기 표시
* 주문취소 표시

#### 결제완료

* 결제하기 숨김
* 주문취소는 정책 확인 후 제한 또는 숨김

#### 배송준비중 / 배송중 / 배송완료

* 결제하기 숨김
* 주문취소 숨김

#### 주문취소

* 취소 버튼 숨김
* 상태 강조 표시

### 프론트 작업

* `getOrderStatusLabel()`
* `canPayOrder()`
* `canCancelOrder()`
  같은 유틸 분리 추천

이렇게 해두면 orders 목록, 상세, admin 모두 재사용 가능하다.

## 4. checkout 후 UX 정리

지금 checkout 자체는 거의 되어 있다.
남은 건 **결제 포함 기준으로 사용자 흐름 정리**다.

### 작업 목표

* 주문하기 클릭 후 로딩 처리
* 중복 클릭 방지
* 주문 생성 성공 후 자연스럽게 결제 단계 진입
* 실패 시 메시지 명확화

### 보강 포인트

* 주문 버튼 로딩 상태
* 생성 중 중복 제출 막기
* 재고 부족 메시지 통일
* 주문 생성 성공 후 토스트/안내문구 추가

예시 흐름:

1. 주문하기 클릭
2. 주문 생성 API 호출
3. 성공 시 상세 페이지 이동
4. 상세에서 결제하기 유도

## 5. 주문 목록 페이지 보강

주문 목록은 이미 구현되어 있으니, 남은 건 **행동 연결성과 가독성**이다.

### 추가 작업

* 상태별 색상 통일
* 카드 클릭 영역 명확화
* 미결제 주문은 `결제대기` 느낌이 보이도록 강조 가능
* 목록에서 바로 상세 이동 UX 개선

### 선택 작업

* 최근 주문 상단 고정
* 빈 상태 문구 개선
* 페이지네이션 또는 무한스크롤은 나중

## 6. admin 프론트는 백엔드 대기 기준으로 정리

이건 바로 다 구현하려고 하면 막힌다.
지금 프론트 팀원이 할 수 있는 건 **연결 준비**까지다.

### 프론트 선작업 범위

* admin API 함수 파일 분리
* 상품/카테고리/배너/footer 타입 정의
* mock 데이터 의존 부분 분리
* 화면에서 submit payload 구조 정리

### 백엔드 스펙 오면 바로 붙일 대상

* 상품 등록/수정
* 카테고리 관리
* footer 관리
* 메인 배너 관리
* 텍스트 배너 관리
* spot item 관리

즉, admin은 지금 당장은 “완성”보다
**API 연결 가능한 구조 정리**가 프론트 역할이다.

# 백엔드에 바로 요청해야 할 것

프론트가 결제를 붙이려면 이것부터 받아야 한다.

1. 결제 API 경로
2. 결제 요청/응답 스펙
3. 결제 성공 시 주문 상태값
4. 취소 가능한 상태 범위
5. 주문 상세 item 가격 필드명
6. 테스트 결제 방식

여기서 5번은 지금 바로 필요한 항목이다.
현재 화면상 `주문가격 0원 / 합계 0원`이 보여서 이건 먼저 맞춰야 한다.

# 한 줄 정리

지금 프론트 팀원의 실제 남은 일은
**“주문 기능 구현” 자체가 아니라, 주문 기능을 결제까지 완성하고 상태별 액션/UI를 다듬는 일”**이다.

즉 우선순위는 이렇게 잡으면 된다:

1. 주문 상세 0원 문제 수정

2. 결제하기 버튼 및 결제 흐름 추가

3. 상태별 버튼 분기 정리

4. checkout/order UX 보강

5. admin API 연결 준비
